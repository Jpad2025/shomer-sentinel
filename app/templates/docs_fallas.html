GLOSARIO DE FALLAS

Referencia para el técnico instalador — fallas comunes y pasos para resolverlas.
Haz clic en cada falla para expandir la solución.

Tracker — Inventario
Escaneo de activo no trajo información de software instalado ▾

Causa: el activo no tiene credenciales configuradas o son incorrectas. El escáner necesita usuario/contraseña para leer software en Windows o Linux.

1Clic en el activo en la tabla → drawer lateral → sección Credenciales.
2Agrega o corrige usuario y contraseña (Windows: cuenta local o dominio; Linux: usuario SSH).
3Haz clic en Re-escanear en el drawer o usa Escaneo profundo.
4Espera 30-90 s y recarga la tabla. Si sigue vacío, verifica que el firewall del equipo permita WMI (Windows, puertos 135/445) o SSH (Linux, puerto 22).
El escaneo no descubre nuevos equipos en la red ▾

Causa: la subred configurada no coincide con la red del cliente, o el router no tiene credenciales SSH.

1Tracker → Configuración: verifica que la Subred sea correcta (ej. 192.168.10.0/24).
2Verifica que el router GL.iNet tenga usuario y contraseña SSH en Guardian → Dispositivos.
3Lanza un Escaneo manual. Revisa logs en Estado → shomer-tools.
Export de Excel o PDF falla o descarga archivo vacío ▾

Causa: no hay activos en el inventario, o el servicio shomer-tools está caído.

1Verifica en Estado del sistema que shomer-tools esté active.
2Si está caído: sudo systemctl restart shomer-tools.
3Confirma que la tabla de activos tenga registros (contador > 0) antes de exportar.
Protector — Backups
La ruta de backup de un equipo no encuentra los archivos ▾

Rutas correctas por sistema operativo:

  • Windows: C:/Users/nombre/Documents (barras normales o dobles invertidas)
  • macOS: /Users/nombre/Documents
  • Linux: /home/nombre/documentos
  • NAS Synology/QNAP: /volume1/shared
1Protector → Dispositivos: abre el equipo y corrige la ruta.
2Usa Probar conexión para verificar acceso.
3Lanza un Backup ahora manual para confirmar.
Sincronización con Backblaze B2 falla — error de credenciales ▾

Causa: Application Key, Key ID o nombre del bucket incorrectos.

1Protector → Configuración B2: verifica Key ID, Application Key y nombre del bucket.
2Usa Probar conexión B2 — muestra el error exacto de la API de Backblaze.
3Si el bucket no existe, créalo en la consola de Backblaze con el mismo nombre configurado.
4Verifica que el Application Key tenga permisos de lectura y escritura sobre ese bucket.
Restic muestra "wrong password" al listar snapshots ▾

Causa: la contraseña del repositorio Restic cambió o no está configurada.

1Verifica RESTIC_PASSWORD o RESTIC_PASSWORD_FILE en /etc/shomer/shomer-runtime.env.
2Si el repositorio es nuevo, inicialízalo: restic -r [RUTA] init.
3Reinicia shomer-tools después de cambiar variables: sudo systemctl restart shomer-tools.
Guardian — Monitoreo de red
El panel no muestra ningún nodo / todos aparecen offline ▾

Causa probable: Redis no corre, o los nodos GL.iNet no son alcanzables por red.

1Estado del sistema: verifica que redis-server y shomer-monitor estén active.
2Si Redis está caído: sudo systemctl restart redis-server, luego reinicia shomer-guardian.
3Guardian → Dispositivos: verifica que los nodos tengan IP alcanzable (prueba con ping [IP]).
4Si hay VLANs: confirma que haya enrutamiento L3 entre el Shomer y los AP.
Las notificaciones de Telegram no llegan ▾

Causa: token del bot o chat ID incorrectos, o el Shomer no tiene acceso a internet.

1Guardian → Configuración: verifica Token y Chat ID.
2Usa el botón Probar Telegram — muestra el error exacto si falla.
3Verifica conectividad a internet: curl -s https://api.telegram.org debe responder.
4Asegúrate de que el bot esté en el grupo/canal con permisos para enviar mensajes.
El reboot automático del GL.iNet falla — error SSH ▾

Causa: la llave SSH del Shomer no está autorizada en el GL.iNet.

1Guardian → Dispositivos: verifica que el router tenga usuario SSH (root) configurado.
2Copia la llave pública: ssh-copy-id -i ~/.ssh/id_rsa_shomer.pub root@[IP-GL.iNet].
3Prueba conexión manual: ssh -i ~/.ssh/id_rsa_shomer root@[IP-GL.iNet].
Hunter — Ciberseguridad
El pipeline muestra Suricata inactivo ▾

Causa: Suricata no corre, posiblemente por error de nombre de interfaz mirror.

1sudo systemctl status suricata — revisa el estado.
2sudo journalctl -u suricata -n 50 — identifica el error exacto.
3Si el error es por interfaz: edita /etc/suricata/suricata.yaml y corrige af-packet → interface con el nombre real (ver Estado → NICs).
4sudo systemctl restart suricata.
Reglas guardadas en Hunter no generan alertas ▾

Causa: Suricata no recargó las reglas. Guardar en el panel no recarga automáticamente.

1Después de guardar reglas, clic en Recargar Suricata en el panel Hunter.
2Si falla: sudo kill -USR2 $(pidof suricata) o sudo systemctl reload suricata.
Sistema general
El panel muestra "502 Bad Gateway" ▾

Causa: nginx corre pero shomer-guardian (puerto 8000) está caído.

1sudo systemctl status shomer-guardian
2Reinicio seguro (evita huérfanos en puerto):
sudo systemctl stop shomer-guardian sudo lsof -ti:8000 | xargs sudo kill -9 2>/dev/null sleep 2 && sudo systemctl start shomer-guardian

Si falla repetidamente: journalctl -u shomer-guardian -n 50.

El panel muestra "401 Unauthorized" en todas las peticiones ▾

Causa: sesión expirada o JWT_SECRET cambió.

1Cierra sesión y vuelve a entrar desde /login.
2Si afecta a todos los usuarios: verifica que JWT_SECRET no haya cambiado en /etc/shomer/shomer-runtime.env.
3Reinicia ambos servicios después de cualquier cambio en ese archivo.
CPU o RAM muy altos — panel responde lento ▾

Causa probable: escaneo profundo o backup en segundo plano.

1Revisa Estado del sistema → métricas CPU/RAM en tiempo real.
2htop en terminal para identificar el proceso.
3Si es nmap bloqueado: sudo pkill nmap. Si es Restic: es normal durante backup, espera.
